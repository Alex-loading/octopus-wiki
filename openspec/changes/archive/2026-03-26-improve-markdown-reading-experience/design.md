## Context

当前文章详情页在 `Post` 页面中使用手写规则解析 Markdown，语法支持不完整，且不同元素的版式节奏不统一。项目已完成文章入库与后台管理闭环，正文以 Markdown 作为主要内容格式，因此渲染层的稳定性、可读性和安全性成为当前主要瓶颈。

约束条件：
- 保持现有页面结构与视觉语言（明暗主题、灰阶与 indigo 强调）一致。
- 不引入新的内容模型或页面路由。
- 优先在现有 React + Vite 架构内做最小侵入替换。

## Goals / Non-Goals

**Goals:**
- 提供稳定、可扩展的 Markdown 渲染链路，覆盖常见写作语法。
- 通过统一 Typography 映射提升长文阅读体验与可扫描性。
- 保持目录、锚点、滚动定位能力并增强一致性。
- 在渲染阶段加入安全清洗，降低注入风险。

**Non-Goals:**
- 不在本次实现完整 CMS 编辑器重构。
- 不在本次引入 MDX 组件化内容编排。
- 不新增与阅读无关的交互功能（推荐算法、评论系统改造等）。

## Decisions

1. 采用 `react-markdown` 作为渲染主引擎，替换手写逐行解析
- 原因：在 React 中可直接将 AST 节点映射到受控组件与现有 Tailwind 样式，维护成本低。
- 备选：`markdown-it`（输出 HTML 再注入）与现有组件体系耦合较弱；`mdx` 引入内容治理复杂度过高。

2. 使用 `remark-gfm` 扩展 GFM 语法支持
- 原因：覆盖表格、任务列表、删除线等常见内容形态，满足运营写作实际需求。
- 备选：仅 CommonMark 会导致现有与未来文章语法受限。

3. 使用 `rehype-sanitize` 进行渲染安全清洗
- 原因：文章内容来源于数据库，必须对不受控 HTML 做白名单约束。
- 备选：完全禁用 HTML 可进一步收紧风险，但会降低兼容性；本次采用 sanitize 兼容安全平衡。

4. 使用 `rehype-slug` 与 `rehype-autolink-headings` 强化标题锚点
- 原因：与现有目录导航与滚动定位能力天然配合，提升长文跳转体验。
- 备选：继续手写 slug 与锚点逻辑，重复维护成本较高。

5. 建立统一 Markdown 元素样式映射（Typography Contract）
- 原因：保证 `h1-h3/p/ul/ol/blockquote/code/pre/hr/table/a` 在明暗主题下具备一致间距、层级与对比度。
- 备选：沿用页面内零散类名会继续导致样式漂移。

## Risks / Trade-offs

- [插件链引入后渲染行为变化] → 通过真实文章样本回归（含代码块/列表/表格）并保留回退分支。
- [sanitize 白名单过严导致展示缺失] → 先采用最小白名单，逐步按内容需求扩展并记录变更。
- [样式映射过重影响首屏性能] → 复用现有 Tailwind 体系，避免额外运行时样式库。
- [标题锚点规则变化影响目录高亮] → 使用统一 slug 规则并回归目录滚动联动。

## Migration Plan

1. 在文章渲染组件中引入 `react-markdown` 与插件链，完成基础节点映射。
2. 用统一样式映射替换散落的逐行样式逻辑，覆盖明暗主题。
3. 对目录锚点与滚动高亮进行联调，确保行为一致。
4. 以现有文章样本做回归验证（尤其代码块、列表、引用、表格）。
5. 若线上出现兼容异常，可临时回退到旧渲染器实现。

## Open Questions

- 是否允许受控范围内的原生 HTML（如 `<kbd>`）在首期被保留？
- 代码块高亮首期是否仅保留基础样式，语法高亮作为后续迭代？
- 是否需要在后台编辑页同步提供“与前台一致”的即时预览？
