## Context

当前项目已经完成文章数据库读取能力，但缺少“内容管理端”来完成上传、删除、修改闭环。现有更新方式偏向开发流程，无法满足日常运营效率与权限安全要求。约束包括：
- 保持现有主站视觉与交互风格一致（复用现有组件与样式 token）。
- 基于既有 Supabase 能力最小增量实现，不引入重型 CMS 平台。
- 前台继续通过内容仓储层读取已发布内容，后台负责写操作与状态管理。

## Goals / Non-Goals

**Goals:**
- 提供可用的文章管理后台：列表、创建、编辑、删除、发布状态切换。
- 建立可控写入链路：字段校验、鉴权、错误处理、操作反馈。
- 确保后台 UI 与主站一致，减少维护分叉。
- 保证前台稳定：写入失败不影响当前已发布内容读取。

**Non-Goals:**
- 不在本次实现完整通用 CMS（多租户、插件系统、复杂工作流）。
- 不在本次引入复杂富文本协同编辑。
- 不在本次覆盖 demo 管理与其他内容类型后台。

## Decisions

1. 采用 Supabase + 自建轻后台（React 页面）
- 方案：在现有应用中新增 `/admin/articles` 管理路由与页面，后台数据直连 Supabase。
- 原因：最小改动、与当前架构一致、上线速度快。
- 备选：引入 Strapi/Directus；功能完整但迁移成本、部署复杂度和 UI 一致性成本更高。

2. 数据模型维持现有 `articles` 主表，补充审计与软删除字段
- 方案：在现有字段基础上补充 `deleted_at`、`updated_by`（如尚未存在），删除行为采用软删除。
- 原因：避免误删导致内容不可恢复，便于审计与回滚。
- 备选：物理删除；实现简单但风险高、恢复困难。

3. 后台写操作采用显式服务层封装
- 方案：新增 `createArticle`、`updateArticle`、`deleteArticle`、`publishArticle` 等写接口，统一处理校验与错误映射。
- 原因：避免页面直写 SQL 风格调用，提升可测试性与可维护性。
- 备选：页面直接调用 Supabase SDK；速度快但逻辑分散。

4. 权限控制使用 Supabase Auth + RLS
- 方案：仅允许 `admin` 角色用户访问管理页面和执行写操作，未授权用户重定向或返回 403。
- 原因：把安全边界前置到数据库层，避免仅靠前端隐藏。
- 备选：仅前端路由守卫；安全性不足。

5. UI 风格与主站一致
- 方案：复用 `src/app/components/ui` 与现有主题变量，不新增独立设计体系。
- 原因：降低认知与维护成本，保证产品一致性。
- 备选：单独后台主题；实现更快但视觉割裂。

## Risks / Trade-offs

- [RLS 规则配置错误导致越权或误拒绝] → 先在预发使用最小权限策略并编写权限验证用例。
- [软删除导致查询复杂度上升] → 在查询层默认过滤 `deleted_at is null` 并增加索引。
- [管理页与主站耦合导致发布风险扩大] → 管理页路由与前台读取逻辑分层，关键功能独立验证。
- [编辑器能力不足影响运营体验] → 首期使用 Markdown + 基础预览，后续按反馈扩展。

## Migration Plan

1. 新增后台路由与空页面骨架，接入认证守卫。
2. 扩展内容仓储写接口并接入 Supabase 写能力。
3. 实现文章列表与创建/编辑表单，完成字段校验。
4. 实现删除（软删除）与发布状态切换。
5. 补充 RLS 与角色策略，完成预发联调。
6. 上线前执行回归：前台读取、后台写入、权限拒绝、异常提示。

回滚策略：若后台写入链路异常，临时关闭 `/admin` 入口并保留前台只读链路；必要时将 `VITE_CONTENT_SOURCE` 切回静态兜底。

## Open Questions

- `admin` 角色来源是独立用户表还是 JWT claim 映射？
- 是否需要“定时发布”作为本期能力，还是仅支持立即发布？
- 删除后是否需要前台立即失效缓存（若后续引入缓存层）？
